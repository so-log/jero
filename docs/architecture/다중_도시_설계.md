# 다중 도시 — 설계문서

> 기획: `docs/planning/17_다중_도시.md`.
> 핵심 원칙: **① 도시를 1급 엔티티로 정규화 ② 기존 단일 도시 여행 무중단 하위호환 ③ 단계별(phase) 구현으로 회귀 최소화.**
>
> **디자인 시안**: `docs/design/prototype/다중 도시.dc.html`(데스크톱 1280 + 모바일 375). 확정 UI:
> - 마법사: "여행 기간" → **"도시·일정" Step**(도시명+박수 추가·드래그, "N개 도시 · 총 N일", 날짜 자동 계산)
> - 플랜/캘린더: **도시 시작일 배지**("4월 11일 · 오사카 첫날"), 도시 전환 시 지도·일정 이동, 캘린더 도시별 색·라벨 구간
> - (Phase 5) 도시 간 이동: 경계에 **이동 카드 자동 배치**("신칸센 노조미 · 4/11 오전 · 약 15분", 기차/항공 + 소요시간)

## 1. 데이터 모델

```
trip 1 ─── N trip_city 1 ─── N place
```

### 1.1 신규 테이블 `trip_city`
```sql
create table public.trip_city (
  id         uuid primary key default gen_random_uuid(),
  trip_id    uuid not null references public.trip(id) on delete cascade,
  name       text not null,          -- 도시명(예: 도쿄)
  country    text,                   -- 국가(예: 일본)
  lat        numeric,                -- 도시 중심(지도 이동용, 선택)
  lng        numeric,
  nights     int not null,           -- 박수
  seq        int not null,           -- 여행 내 순서(0-based)
  created_at timestamptz not null default now(),
  unique (trip_id, seq)
);
create index trip_city_trip_idx on public.trip_city(trip_id);
```

### 1.2 `place`에 `city_id` 추가 (additive)
```sql
alter table public.place add column city_id uuid references public.trip_city(id) on delete set null;
create index place_city_idx on public.place(city_id);
```
- nullable → 하위호환(기존 place는 백필로 채움, 신규는 배정).
- `on delete set null` → 도시 삭제 시 장소는 미배정으로 남음(삭제 아님, B3 패턴).

### 1.3 날짜 파생 (도시 → 날짜 범위)
- 여행 `start_date` 기준, 도시들을 `seq` 순으로 `nights`만큼 누적해 각 도시의 `[startDay, endDay)` 범위 계산(순수 함수 `lib/trip/citySchedule.ts`).
- **Day → 도시**: 어떤 날짜가 속한 도시 = 그 날짜를 포함하는 범위. 날짜는 **파생**이라 도시엔 날짜를 저장하지 않음(박수·순서만 = 단일 출처, 박수 변경 시 자동 재계산).
- 정합: `Σ nights = 여행 총 박수`. (마지막 도시 endDay = 여행 end_date.)

## 2. 하위호환 (핵심)

- **마이그레이션 백필**: 기존 각 trip에 대해 `trip.country/region` 으로 **도시 1개**(`seq=0`, `nights=여행 박수`) 생성 → 그 trip의 모든 place에 `city_id` 설정.
- **앱 동작**: 항상 `cities[]`를 다루되, **길이 1이면 현재 UX와 동일**(도시 전환 UI 숨김, 지도/플랜 그대로). 다중 도시 UI는 `cities.length > 1`에서만 노출.
- `trip.country/region`은 당분간 유지(호환·표시). 장기적으로 첫 도시로 대체 가능(별도 정리).

## 3. RPC · 계약

### 3.1 `create_trip` 확장
- payload에 `cities: [{ name, country, nights, lat?, lng? }]` 추가.
- 트랜잭션: trip insert → cities를 seq 순 insert → (템플릿이면 place 복제 시 city 매핑). cities 미제공(구버전 호출)이면 country/region으로 도시 1개 생성(폴백).
- `start_date`/`end_date`는 박수 합과 정합(마법사가 계산해 전달, 서버 재검증).

### 3.2 신규 훅
- `useTripCities(tripId)` — 도시 목록(seq 순) 조회. 파생 날짜 범위 포함.
- `useUpsertCity` / `useReorderCities` / `useDeleteCity` — CRUD·순서(editor+). 순서/박수 변경 시 `['places', tripId]`·`['trip', tripId]` 무효화.
- 기존 `usePlacesQuery` 응답에 `cities[]` 추가(additive) 또는 별도 쿼리.

### 3.3 RLS (0002 패턴 재사용)
```sql
alter table public.trip_city enable row level security;
create policy city_select on public.trip_city for select using ( public.is_trip_member(trip_id) );
create policy city_write  on public.trip_city for all
  using ( public.trip_role(trip_id) in ('owner','editor') )
  with check ( public.trip_role(trip_id) in ('owner','editor') );
```

## 4. 셀렉터 (순수, `features/*/lib`)
- `citySchedule(cities, tripStart)` → `[{cityId, name, startDate, endDate, dayCount}]`.
- `cityForDate(schedule, date)` → 그 날짜의 도시.
- `placesForCity(places, cityId)` / `placesForDay(...)`(기존)에 도시 필터 결합.

## 5. 뷰별 영향 · 단계(phase)

- **Phase 1 — 스키마·계약·하위호환 (UI 무변경)**: `trip_city` + `place.city_id` + 마이그레이션(백필) + `create_trip` 폴백 + 셀렉터·쿼리. 앱은 도시 1개로 기존과 동일 동작. **회귀 0 검증이 목표.**
- **Phase 2 — 생성 마법사 다중 도시 입력**: Step1(또는 신규 Step)에서 도시 추가(이름·박수, 순서 드래그) → 날짜 자동 배분·정합 검증 → `create_trip(cities)`.
- **Phase 3 — 플랜/캘린더 도시 컨텍스트**: Day 스위처에 도시 라벨·경계, 지도 도시 중심 이동, 플랜/동선은 그 도시 장소만. 캘린더는 도시별 색/구분.
- **Phase 4 — 장소 뷰 도시 축**: 장소를 도시별 그룹/필터(폴더·카테고리에 이어 3번째 축 — U2 IA 교훈 반영해 라벨 명확히). 장소 저장 시 현재 도시 기본 배정.
- **Phase 5 (선택) — 도시 간 이동·통계**: 세그먼트 경계 이동(항공/기차) 표현, 통계 도시별 분해, 예산 도시 태그.

> 각 phase는 독립 브랜치·PR. Phase 1이 하위호환을 보장하므로 이후 phase는 점진 적용 가능.

## 6. 기존 기능과의 상호작용
- **동선 최적화**: 하루(=한 도시 내) 단위라 무충돌. 도시 간 최적화는 범위 밖(이동수단 다름).
- **B3 여행 날짜 수정**: 도시 박수 변경과 일관되게 — 범위 밖 배정 장소는 미배정 처리(기존 ConfirmDialog 패턴 재사용).
- **selectionStore(플랜↔캘린더 날짜 동기화)**: 선택 날짜 → 도시 파생이라 자연 연동.
- **반응형**: 도시 전환 UI도 모바일 대응(드로어/세그먼트 패턴 재사용).

## 7. 지도
- 도시별 `lat/lng` 중심 → 현재 도시로 `flyTo`(기존 2단계 인프라 재사용). 도시 좌표는 생성 시 지오코딩 or 대표 좌표.
- 여러 도시를 한 지도에 동시 표시는 하지 않음(거리 문제) — 도시 단위 뷰.

## 8. 테스트
- **순수**: `citySchedule`(박수→날짜 범위, 합 정합), `cityForDate`, 도시 순서 변경 시 재계산.
- **마이그레이션/백필**: 기존 trip → 도시 1개·place city_id 채움 검증(로컬 Supabase).
- **하위호환**: 단일 도시 여행이 기존 e2e(flows·budget 등) 그대로 통과(회귀 0).
- **통합**: create_trip(cities) → 도시·날짜 범위·place 매핑, viewer 편집 거부.

## 9. 리스크 · 완화
- **회귀 위험(광범위 영향)** → Phase 1에서 하위호환·기존 e2e 통과를 게이트로. UI는 `cities.length>1`에서만 분기.
- **계약 변경** → `데이터모델_계약.md`에 trip_city 반영, 응답 예시=테스트 fixture 재사용(§7.2).
- **날짜 정합 복잡도** → 날짜를 저장하지 않고 박수·순서에서 파생(단일 출처)해 불일치 원천 차단.

## 10. 구현 순서 요약
1. Phase 1: 마이그레이션(`0006_multicity.sql`) + 셀렉터 + create_trip 폴백 + 하위호환 e2e (브랜치 `feat/multicity-schema`).
2. Phase 2: 마법사 도시 입력.
3. Phase 3: 플랜/캘린더 도시 컨텍스트.
4. Phase 4: 장소 도시 축.
5. Phase 5(선택): 이동·통계·예산 도시.

> 착수는 브랜치 → PR, main 직행 금지(CLAUDE.md §7). Phase 1은 "회귀 0"이 성공 기준.

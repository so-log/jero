# 동선 최적화 — 설계문서

> 기획: `docs/planning/16_동선_최적화.md`. 이 문서는 알고리즘·데이터·API·연동·테스트를 다룬다.
> 핵심: **순수 함수로 최적 순서 계산 → 기존 `reorder_places` 로 저장.** 신규 스키마 없음, 기존 인프라 재사용.

## 1. 전체 구조 (의존 방향)
```
PlanView(플랜 뷰)
  └ RouteOptimizeButton / RouteOptimizePreview   (features/itinerary/components)
       ├ useRouteOptimize()      (features/itinerary/hooks) — 계산·미리보기·적용 오케스트레이션
       │    ├ lib/route/costMatrix.ts   — 거리/이동시간 매트릭스 (Haversine | Directions)
       │    └ lib/route/optimize.ts     — NN + 2-opt 순수 알고리즘
       └ useReorderPlaces()      (기존) — 적용 시 order_in_day 저장(낙관적 + reorder_places RPC)
```
- **표현/도메인 분리**: 알고리즘은 `lib/route/`의 **순수 함수**(좌표 배열만 받음, 도메인/DOM 무지). 컴포넌트 직접 fetch 금지 — 데이터는 훅 경유.

## 2. 알고리즘

### 2.1 비용 매트릭스 (`lib/route/costMatrix.ts`)
n개 장소 간 이동 비용 `cost[i][j]`. 두 소스:
- **(A) Haversine 직선거리** — 무료·즉시·오프라인. 기본값이자 폴백. (기존 `stats.ts`의 Haversine 재사용/승격.)
- **(B) Google 실이동시간** — `DistanceMatrixService`(도보/대중교통 mode). 정확하지만 유료·쿼터.
- **전략**: 기본은 (A)로 즉시 최적화 → 사용자가 "실이동시간" 옵션 켜면 (B) 1회 호출로 매트릭스 채움. (B) 실패 시 (A) 폴백.

### 2.2 최적화 (`lib/route/optimize.ts`)
하루 방문지는 보통 3~12곳(소규모) → TSP 완전탐색(n!)은 과함. **휴리스틱**으로 충분:
1. **Nearest-Neighbor**: 시작점에서 가장 가까운 곳을 반복 선택해 초기 경로 구성. O(n²).
2. **2-opt 개선**: 경로의 두 간선을 뒤집어(swap) 총 비용이 줄면 채택, 개선 없을 때까지 반복. 지역 최적해로 수렴. O(n²) × 반복(소규모라 저렴).
- **결과 보장**: 2-opt는 항상 초기해보다 총 비용이 **작거나 같음** → 수용 기준(총 이동 ≤ 원래) 충족.

```ts
// 시그니처(순수)
export function optimizeRoute(
  cost: number[][],              // n×n 비용 매트릭스
  opts?: { start?: number; end?: number }, // 앵커(고정 인덱스)
): number[]                      // 최적 방문 순서(인덱스 배열)
```

### 2.3 앵커(시작/끝 고정)
- `start`/`end` 지정 시 해당 노드를 고정하고 **중간 노드만** NN+2-opt. 2-opt swap 시 앵커 위치 불변 제약.
- open path(시작만) / fixed both ends(숙소 왕복) 지원.

## 3. 데이터 계약
- **입력**: 그 Day 장소의 `{id, lat, lng}`(좌표 있는 것만). `usePlacesQuery` 셀렉터로 추출(신규 셀렉터 `scheduledWithCoords(day)`).
- **출력**: 재배치된 `id[]` → 기존 `reorder_places(p_trip_id, p_date, p_ids)` RPC 그대로 사용. **스키마 변경 없음.**
- 좌표 없는 장소는 최적화 제외 후 **끝에 원래 상대순서로 유지**(id 배열 합칠 때 뒤에 append).

## 4. 상태 · 훅 (`useRouteOptimize`)
- `preview(day)`: 매트릭스 계산 → optimizeRoute → `{ order: id[], before: number, after: number }` 반환(적용 안 함).
- `apply(order)`: `useReorderPlaces` 호출(낙관적 setQueryData + RPC) → 리스트·지도·selectionStore 연동 그대로.
- `undo()`: 직전 순서 스냅샷 복원(로컬 보관 후 reorder).
- 계산은 동기(직선거리) 또는 async(Directions). 컴포넌트는 훅만 사용.

## 5. API 사용량 · 비용
- 직선거리 모드: **API 호출 0**(클라 계산). 기본값이라 대부분 무료.
- 실이동시간 모드: DistanceMatrix **1회 요청/최적화**(n×n 한 번). 요청 수 최소화(장소 추가마다 X, 버튼 누를 때만). 키 제한(referrer)·쿼터 유지. Distance Matrix API 콘솔 활성 필요.

## 6. UI 연동 (기존 재사용)
- **번호 마커·Polyline**: 기존 `TripMap`/`RouteLine`/`NumberedMarker` 그대로 — 미리보기는 순서 배열만 바꿔 렌더.
- **이동시간 배지(선택)**: 카드 사이 또는 마커 간 이동시간 표시(실이동시간 모드).
- **총계 배너**: "총 이동 3.2h → 1.8h" — before/after.
- 수동 드래그(dnd-kit)와 공존: 최적화는 순서를 제안·적용할 뿐, 이후 수동 드래그 가능.

## 7. 테스트 (데이터→렌더 우선)
- **순수 함수 유닛**(`optimize.test.ts`): 고정 좌표 세트 → 기대 순서; 2-opt가 초기해보다 총비용 감소; 앵커 고정 검증; n<2 처리.
- **costMatrix.test.ts**: Haversine 대칭성·자기자신 0; Directions 실패 시 폴백.
- **훅/컴포넌트 통합**: preview→apply가 `reorder_places` 호출 payload 정확(모킹), viewer 비노출.
- **e2e(선택)**: 플랜에서 최적화 버튼 → 순서 변경 반영(실 Supabase).

## 8. 성능
- n ≤ ~12 가정. NN O(n²) + 2-opt O(n²)×반복(소규모라 수 ms). 큰 n(>20) 대비 2-opt 반복 상한(guard) 둠.
- 직선거리 모드는 완전 클라 계산이라 즉시 반응.

## 9. 트리플 대비 우위 (구현으로 증명)
- **알고리즘**: 근접순(그리디)이 아니라 **2-opt 지역최적** → 실제로 더 짧은 동선.
- **실이동시간**: 직선이 아닌 도보/대중교통 실제 시간 반영(옵션).
- **제약**: 숙소 시작/복귀 앵커.
- **투명성**: 전/후 총 이동 비교 제시.
- 포트폴리오 관점: "CRUD 너머 알고리즘·최적화·외부 API·비용 관리"를 한 기능에서 보여줌.

## 10. 구현 순서(제안)
1. `lib/route/optimize.ts`(+테스트) — 순수 알고리즘부터(TDD).
2. `lib/route/costMatrix.ts`(Haversine 먼저, Directions 옵션 뒤).
3. `useRouteOptimize` 훅 + 셀렉터.
4. UI(버튼·미리보기·총계 배너) — 기존 지도/리스트 재사용.
5. 실이동시간(Directions) 옵션 + 폴백.
6. 검증(유닛→통합→e2e) + `yarn check`.

# 여행 플래너 — 화면 구성 요구사항 v0.2

> 기능 명세서 기반의 **화면 요구사항 문서**.
> 이 문서는 "각 화면에 무엇이 들어가야 하는가"만 정의한다.
> **배치·레이아웃·색상·타이포·구체적 스타일은 정하지 않는다** — 디자이너(클로드 디자인)가
> 상업용 제품 기준으로 자유롭게 결정하도록 여백을 남긴다.

작성일: 2026-06-23 · 상태: 디자인 생성용

---

## 0. 디자인에 위임하는 것 (정하지 않음)

아래는 **의도적으로 정하지 않는다.** 디자이너가 제품 성격에 맞게 결정한다.

- 색상 팔레트 / 브랜드 컬러 / 다크·라이트
- 레이아웃·그리드·영역 배치 (몇 분할인지, 사이드바인지 탑바인지 등)
- 타이포그래피, 간격, 모서리·그림자 스타일
- 컴포넌트의 시각적 형태 (카드 모양, 버튼 스타일 등)
- 일러스트·이미지 스타일

이 문서가 정하는 것은 **"화면에 어떤 정보와 기능 요소가 존재해야 하는가"**, 그리고 **제약·원칙**뿐이다.

---

## 1. 제품 정의

- **무엇**: 친구들과 함께 여행 일정을 짜고, 장소 순서·동선·예산을 관리하는 **협업 여행 플래너 (웹 서비스)**
- **성격**: 실제 배포·운영하는 **상업용 SaaS 제품** 수준의 완성도. 가입한 사용자가 매일 쓸 수 있는 도구.
- **핵심 가치**: 위치 저장에 그치지 않고 순서·동선·예산·역할까지 여러 명이 함께 굴린다.

---

## 2. 기술 / 플랫폼 전제

- **데스크톱 웹 우선** (기준 폭 1280px)
- **반응형**: 태블릿 / 모바일까지 대응
- 한국어 기본 (다국어 확장 여지)
- 상업 제품이므로: 빈 상태·로딩·에러·권한 제한 등 **모든 상태가 설계되어 있어야 함**

---

## 3. 페이지 목록 (라우트 단위)

총 **6개 라우트** + 워크스페이스 내부 **4개 뷰** + 오버레이 3종.

| # | 페이지 | 목적 |
|---|---|---|
| 1 | 로그인 / 랜딩 | 진입·인증, 제품 소개 |
| 2 | 내 여행 목록 (홈) | 내가 속한 여행들 한눈에, 새 여행 시작 |
| 3 | 여행 생성 (다단계) | 새 여행의 기본 정보·멤버·시작방식 입력 |
| 4 | 여행 워크스페이스 ★ | 한 여행을 작업하는 핵심 공간 (아래 4개 뷰) |
| 5 | 뷰어 공유 (읽기 전용) | 비편집자/외부인에게 일정만 공유 |
| 6 | 계정 설정 | 프로필·기본 설정 (간단) |

**워크스페이스 내부 뷰 (4):**
- 플랜 뷰 — 지도 + 일정(순서/동선)
- 일정표 뷰 — 월/주/일
- 장소 뷰 — 저장 장소·폴더
- 예산 뷰 — 지출·정산 대시보드

**오버레이 (모달/패널, 독립 페이지 아님):**
- 장소 상세 · 멤버/공유 관리 · 지출 추가

---

## 4. 페이지별 화면 구성 요소

> 각 항목은 "이 요소가 화면에 존재해야 한다"는 요구사항. **배치 순서·위치는 디자인에 위임.**

### 4.1 로그인 / 랜딩
- 제품명·한 줄 가치 제안
- 소셜 로그인(구글) / 이메일 로그인·회원가입
- 공유 링크로 둘러보기(비로그인 열람) 진입점
- (선택) 핵심 기능 소개 섹션 — 상업 제품 랜딩 느낌

### 4.2 내 여행 목록 (홈)
- 전역 헤더: 제품명, 검색, 사용자 메뉴
- "새 여행 만들기" 액션
- 여행 항목들의 목록 — 각 항목에 필요한 정보: 커버/대표 이미지, 제목, 기간, 함께하는 멤버, 내 역할
- 필터/구분: 예정·지난 여행, 정렬
- 빈 상태: 첫 여행 만들기 유도

### 4.3 여행 생성 (다단계 입력)
- 진행 단계 표시
- 입력 항목: 제목, 대표 아이콘/커버, 나라, 지역, 기간(시작~종료)
- 멤버 초대: 이메일/링크, 각 멤버 권한(편집/읽기) 지정
- 시작 방식 선택: 빈 여행 / 템플릿 복제
- 단계별 검증·진행, 생성 완료 후 워크스페이스로 이동

### 4.4 ★ 여행 워크스페이스 (공통)
모든 뷰에서 유지되는 요소:
- 여행 식별: 제목, 대표 아이콘, 기간
- 현재 접속 중인 멤버 표시 (실시간 presence)
- 4개 뷰 간 전환 수단
- 공유 진입점
- 목록으로 돌아가기

#### 4.4.a 플랜 뷰 (지도 + 일정) — 핵심
- 지도: 장소 마커, 일정에 등록된 장소는 **순서 번호 + 순서대로 이어지는 동선**, 저장만 된 장소는 구분되는 표시
- 일정 영역: 선택한 날짜의 장소들을 **순서대로** 나열, **드래그로 순서 변경 가능**, 순서 변경 시 지도 동선 즉시 갱신
- 날짜 전환 수단
- 장소 추가 진입점
- 필터: "오늘(선택일) 일정만 보기" 등 지도 표시 토글, 폴더·카테고리 필터
- 실시간: 다른 멤버가 편집 중인 항목·커서 식별

#### 4.4.b 일정표 뷰
- 월 / 주 / 일 보기 전환
- 월: 날짜별 일정 유무·개수 파악 가능
- 주: 여러 날을 나란히
- 일: 시간 순 타임라인, 장소별 **카테고리 구분 아이콘**, 메모 표시
- 긴 목록도 매끄럽게(가상화)

#### 4.4.c 장소 뷰
- 폴더 목록·관리(추가)
- 폴더별 저장 장소 목록 — 장소명, 위치, 카테고리, 메모
- 장소를 "특정 날짜·순서로 일정에 등록" 하는 수단
- 빈 상태: 첫 장소 저장 유도

#### 4.4.d 예산 뷰 (대시보드)
- 요약 지표: 총지출, 1인당, 남은 예산(설정 시), 최다 지출 항목 등
- 카테고리별 지출 비중 차트
- 일자별 지출 추이 차트
- **정산 요약**: 누가 누구에게 얼마(더치페이 계산)
- 지출 내역 목록(테이블): 날짜·항목·금액·결제자·분담 인원
- 지출 추가 진입점

### 4.5 뷰어 공유 (읽기 전용)
- 플랜/일정표와 같은 정보를 보여주되 **편집·추가·관리 요소는 모두 제거**
- "읽기 전용으로 보는 중" 안내 + 로그인하고 편집 유도

### 4.6 계정 설정
- 프로필(이름·이미지), 로그아웃
- (확장) 알림·기본 통화 등

---

## 5. 공통 컴포넌트 (기능 단위 — 시각 형태는 위임)

- 여행 항목 표현 / 장소 항목 표현(드래그 가능)
- 카테고리 표시(식당·카페·기념품·쇼핑·박물관·숙소·기타 등 구분)
- 멤버 표현 / 접속 멤버 묶음 표시
- 역할 표시(소유자·편집자·뷰어)
- 필터·토글
- 뷰 전환 컨트롤
- 다단계 입력의 단계 표시
- 지표 요약 / 차트
- 빈 상태 안내

---

## 6. 상태 요구사항 (전 화면 공통)

- **빈 상태**: 안내 + 다음 행동 유도
- **로딩**: 콘텐츠 로딩 중 처리
- **에러**: 네트워크·권한 오류 처리
- **권한별**: 뷰어는 편집/추가/관리 요소 비노출 또는 비활성
- **실시간**: 동시 편집 반영, 접속 멤버·편집 위치 표시

---

## 7. 권한 분기

| 요소 | 소유자 | 편집자 | 뷰어 |
|---|---|---|---|
| 일정·지도·예산 보기 | O | O | O |
| 장소/순서/메모/지출 편집 | O | O | ✕ |
| 멤버·권한 관리 | O | ✕ | ✕ |
| 여행 삭제 | O | ✕ | ✕ |

## 8. 아이콘 & 비주얼 원칙

### 8.1 분위기 / 톤 (가장 중요)
**밝고 가벼우면서 아주 깔끔한 느낌.** iOS(아이폰) 캘린더처럼 정돈되고 군더더기 없는 스타일을 지향.

- 밝은 배경, 흰색/연한 회색 카드, 둥근 모서리, 부드럽고 은은한 그림자
- 넉넉한 여백과 명확한 시각적 위계 (정보가 빽빽하지 않게)
- 부드러운/파스텔 톤의 포인트 컬러, 카테고리·태그는 연한 컬러 칩으로 구분
- 세그먼트(알약형) 토글, 둥근 버튼 등 모던하고 친근한 컴포넌트
- 정보 밀도가 높은 영역(지도·일정·대시보드)도 이 깔끔한 톤을 유지하며 **가독성 우선**
- 전체적으로 **상업용 SaaS 제품 수준**의 신뢰감과 완성도

> 정확한 색 팔레트·간격·타이포는 이 톤을 만족하는 선에서 **디자이너가 자유롭게** 결정한다.

### 8.2 아이콘
- **이모지 사용 금지.** (시스템 이모지 X)
- 아이콘은 **새로 제작하거나, 라이선스 허용되는 무료 아이콘 세트**(예: Lucide, Tabler, Heroicons 등)를 **하나 골라 일관되게** 사용. 카테고리 아이콘은 한 세트 안에서 통일된 스타일로.
- 카테고리별로 **서로 구분되는** 아이콘이 필요(식당/카페/기념품/쇼핑/박물관/숙소/기타). 단, 어떤 모양·색인지는 디자이너 결정.
- 실시간 협업을 위해 **멤버를 서로 구분하는 식별 수단**(색 등)이 필요 — 구체적 색은 디자이너 결정.

> 색상·스타일·레이아웃은 위 원칙을 만족하는 선에서 **디자이너가 자유롭게** 정한다.

## 9. 반응형 요구

- 데스크톱(1280 기준)에서 정보가 충분히 펼쳐지고, 화면이 좁아지면 자연스럽게 재구성된다.
- 워크스페이스의 지도·일정·패널이 좁은 화면에서 어떻게 합쳐지거나 전환될지는 디자이너가 결정.
- 모바일에서도 핵심(일정 보기·순서·예산 확인)이 가능해야 함. 본격 모바일 기록 기능은 향후 네이티브 앱.

---

## 10. 클로드 디자인에 넘길 때

화면 하나씩 생성하되, 매번 함께 전달할 것:
- 해당 화면의 **구성 요소 목록(4장)** + **상태 요구(6장)** + **권한 분기(7장)**
- **비주얼 원칙(8장)**: 이모지 금지, 일관된 무료/자체 아이콘, 상업용 톤, 색·레이아웃은 위임
- "배치·색·스타일은 제품에 맞게 알아서 결정해줘"라고 명시

권장 생성 순서: 플랜 뷰 → 내 여행 목록 → 예산 대시보드 → 일정표 → 생성/장소/오버레이 → 로그인/뷰어.
(처음 만든 화면의 톤·아이콘·컴포넌트를 이후 화면에서 재사용하도록 지시 → 일관성 확보.)

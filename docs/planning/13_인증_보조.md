# 13. 인증 보조 (비밀번호 재설정 · 이메일 인증 · 콜백)

> 로그인(01) 본 흐름을 보조하는 인증 화면 묶음. 백엔드 연동(Supabase Auth)으로 새로 필요해진 화면 — 계약 인증 설계(데이터모델_계약 B3)와 연결된다. 10 오버레이처럼 **여러 하위 화면을 한 문서**로 묶는다.

## 1. 개요
- **화면 ID**: `auth-aux`
- **하위 화면 / 라우트**:
  - **A. 비밀번호 재설정 요청** — `/auth/reset` (이메일 입력 → 재설정 메일 발송)
  - **B. 새 비밀번호 설정** — `/auth/reset`(복구 세션 진입 시) 또는 `/auth/reset?stage=update`
  - **C. 이메일 인증 안내 / 재전송** — `/auth/verify` (가입 후 확인 메일 안내·재전송)
  - **D. 인증 콜백(비 UI)** — `/auth/callback` (OAuth·이메일 확인·복구 링크의 코드 교환 → 세션 수립 후 리다이렉트)
- **목적**: 계정 접근 회복(비번 분실)과 이메일 소유 확인을 안전하게 처리하고, 링크 만료를 우아하게 안내.

## 2. 사용자 & 진입 경로
- **A**: 01 로그인의 "비밀번호를 잊으셨나요?" → `/auth/reset`.
- **B**: 재설정 메일의 링크 클릭 → `/auth/callback`이 복구 세션 수립 → 새 비번 설정 폼.
- **C**: 이메일+비번 회원가입 직후(이메일 확인 필요 시) 또는 미인증 로그인 시도 → 안내 화면.
- **D**: 구글 OAuth 완료, 이메일 확인 링크, 복구 링크 — 모두 이 콜백으로 돌아옴(사용자에겐 잠깐의 처리 화면).

## 3. 구성 요소
> 시안: `docs/design/prototype/인증 보조 화면.dc.html` — **01 로그인과 동일한 좌우 분할**(좌: 브랜드+기능 3종, 우: 상태별 pane 380px). 상태 스위처: `reset`/`sent`/`newpw`/`verify`/`expired`. 공통: 아이콘 배지 + 헤더(제목·설명) + 폼/안내 + 하단 링크. 우측 로딩 오버레이(스피너 + 문구).

- **A 재설정 요청(`reset`)**: `key` 배지 + "비밀번호를 잊으셨나요?" + 이메일 입력 + "재설정 링크 보내기"(send) + "로그인으로 돌아가기".
- **발송 완료(`sent`)**: `mail-check` 배지 + "메일함을 확인하세요"(링크 30분 유효) + 메일 칩 + 재전송 row("다시 보내기"·쿨다운) + 로그인 돌아가기.
- **B 새 비밀번호(`newpw`)**: `lock-open` 배지 + 새 비번 + 확인(보기 토글) + **규칙 체크(8자 이상 · 일치)** + "비밀번호 변경하고 로그인".
- **C 이메일 인증(`verify`)**: `mail` 배지 + "이메일을 확인해 주세요"(가입 마무리) + 메일 칩 + "메일 앱 열기" + 재전송 + "다른 이메일로 가입하기".
- **공통 만료(`expired`)**: 11 시스템 톤 — clock 배지 + "만료됨·EXPIRED" + "링크가 만료되었어요" + "새 링크 요청하기"(→`reset`) + 로그인 돌아가기.
- **D 콜백(비 UI, `/auth/callback`)**: 코드 교환 처리 스피너 후 리다이렉트(실패 시 로그인/만료).

## 4. 데이터
- 화면 폼: `authSchema` 재사용(이메일·비밀번호 규칙, 클라 검증). 서버는 Supabase Auth가 처리.
- **비밀번호 정책(확정)**: 가입·**재설정 새 비밀번호는 8자 이상**(시안 `newpw` 규칙). 로그인은 하드 min 없이 "필수"만(기존 짧은 비밀번호 계정 거부 방지). 계약 B3와 동일 — `authSchema` 반영은 구현 1단계.
- **Supabase Auth 매핑(B3)**:
  - A 요청 → `auth.resetPasswordForEmail(email, { redirectTo: /auth/callback })`
  - B 설정 → (복구 세션 상태에서) `auth.updateUser({ password })`
  - C 재전송 → `auth.resend({ type:'signup', email })`
  - D 콜백 → `auth.exchangeCodeForSession(code)` (또는 `verifyOtp`) → 목적지 리다이렉트
- 민감정보(토큰·코드)는 URL 처리 후 히스토리에 남기지 않도록 교체(replace) — 로깅 금지(§8.5).

## 5. 상태 (시안 스위처 대응)
- **`reset`**: 이메일 입력 / 검증 에러("올바른 이메일…") / 발송 중(오버레이 "메일을 보내는 중…") → `sent`.
- **`sent`**: 발송 완료 안내(30분 유효) + 재전송(쿨다운).
- **`newpw`**: 유효 복구 세션에서 폼 / 규칙 미충족 에러(8자·일치) / 변경 중("비밀번호를 변경하는 중…") / 성공→로그인.
- **`verify`**: 인증 안내 + 재전송(쿨다운). 인증 완료 시 정상 로그인 흐름.
- **`expired`**: 만료/무효 링크 일반화 에러(재설정·확인 공통).
- **`/auth/callback`(비 UI)**: 처리 중 / 성공(리다이렉트) / 실패(만료·무효→에러).
- 공통: 로딩 오버레이, **일반화 에러**(계정 존재 여부 비노출, §8.5).

## 6. 인터랙션
1. **A**: 이메일 입력 → 발송 → **항상 동일한 "메일을 보냈습니다" 안내**(계정 존재 여부 비노출, §8.5) → 재전송은 쿨다운.
2. **B**: 콜백으로 복구 세션 수립 → 새 비번 2회 입력(일치·규칙 검증) → `updateUser` → 성공 시 로그인 세션으로 `/trips` 또는 01.
3. **C**: 안내 → "다시 보내기"(쿨다운 타이머) → 확인 후 인증되면 정상 로그인 흐름.
4. **D**: 코드 교환 성공 → `returnTo` 또는 `/trips` · 실패(만료/무효) → 01 또는 만료 에러.
5. 링크 만료 공통 → "재설정 메일 다시 요청"(A로) / "로그인으로".

## 7. 권한 분기
- 인증 이전 화면이라 역할(owner/editor/viewer) 개념 없음.
- **B 새 비번**은 유효한 **복구 세션**에서만 접근 — 세션 없이 직접 진입 시 만료/무효 에러(서버·Supabase가 강제).
- 세션·토큰 유효성은 서버(Supabase Auth)에서 강제, UI는 표시만(§8.2).

## 8. 라우팅 & 연결
- **라우트**: `/auth/reset`, `/auth/verify`, `/auth/callback` — 모두 **미들웨어 보호 예외**(비로그인 접근 허용, B3).
- **들어오는 곳**: 01 로그인("비밀번호 잊음"), 가입 직후, 이메일 링크(재설정·확인), OAuth 리다이렉트.
- **나가는 곳**: 성공 → `/trips`(또는 `returnTo`) · 실패/만료 → `/`(01) · A 발송 후 → 안내 상태 유지.
- **연계**: 01 로그인/가입(`useAuth`), 계약 B3(Auth·콜백·미들웨어).

## 9. 구현 메모
- `app/auth/callback/route.ts`(Route Handler, 신규) — `exchangeCodeForSession` 후 `redirect`. UI 아님.
- `app/auth/reset/page.tsx`, `app/auth/verify/page.tsx`(신규) — 01 인증 패널 컴포넌트/토큰 재사용(`features/auth`).
- **쿨다운·rate limit**: 재전송 버튼 클라 쿨다운 + 서버(Supabase) rate limit 신뢰(§8.7).
- **세션 쿠키**: 복구/OAuth 세션 모두 HttpOnly+Secure+SameSite 쿠키(B3) — 토큰 localStorage 금지.
- **에러 일반화**: 재설정 요청은 계정 유무와 무관하게 동일 응답(열거 공격 방지, §8.5).
- **디자인 토큰**: 01 우측 패널 폼·버튼·입력(`components/ui/input`·`button`) 재사용. 하드코딩 금지(§7.1).
- **접근성**: 폼 라벨·에러 aria, 비번 보기 토글 aria-label, 콜백 처리 aria-live.

## 11. 수용 기준 (Acceptance Criteria)
> 이 화면이 "done"인 조건. GATE 1 승인·QA 테스트 시나리오의 근거.

- [ ] `/auth/reset`에서 이메일 제출 시 **계정 존재 여부와 무관하게 동일한 발송 안내**가 표시된다(열거 방지).
- [ ] 재설정 메일 링크로 진입하면 새 비밀번호 설정 폼이 열리고, 규칙·일치 검증 후 변경되어 로그인 상태가 된다.
- [ ] 만료·무효 링크(재설정/확인)는 만료 에러와 "다시 요청"/"로그인으로" 액션이 표시된다.
- [ ] 가입 후 이메일 인증 안내가 표시되고, "다시 보내기"가 쿨다운과 함께 동작한다.
- [ ] `/auth/callback`이 OAuth·이메일 확인·복구 코드를 교환해 세션을 수립하고 목적지로 리다이렉트한다(실패 시 로그인).
- [ ] 세션 쿠키는 HttpOnly+Secure+SameSite로 발급되고, 토큰이 URL 히스토리·로그에 남지 않는다(§8.4·§8.5).
- [ ] 새 비밀번호 화면은 유효한 복구 세션 없이는 접근·제출이 거부된다.

**비기능**: 데스크톱 1280px + 모바일 반응형(메일 링크는 모바일 유입) / 기존 토큰 계승 / 일반화 에러·rate limit·로딩 / 폼 접근성.

## 12. 확정 사항 (MVP 범위)
- Supabase Auth 위임(`resetPasswordForEmail`·`updateUser`·`resend`·`exchangeCodeForSession`) — 자체 토큰/메일 인프라 없음.
- 콜백 단일 진입점(`/auth/callback`)으로 OAuth·확인·복구 통합.
- 재설정 요청은 열거 방지 동일 응답. 재전송 쿨다운.
- 이메일 발송은 **Supabase Auth 기본 메일러**(커스텀 템플릿·도메인은 후속).

# 11. 시스템 페이지 (404 / 에러 / 403 / 점검)

> 앱이 정상 흐름을 벗어났을 때 보여주는 전역 안전망. 동일한 중앙 일러스트 레이아웃에 상태별 톤·문구·액션만 바뀐다.

## 1. 개요
- **화면 ID**: `system-pages`
- **표시 방식**: 라우트가 아닌 **전역 폴백** — Next.js 특수 파일(`not-found.tsx`·`error.tsx`·`global-error.tsx`) + 점검 페이지로 구현. 단일 라우트 없음.
- **목적**: 없는 페이지·오류·권한 없음·점검 상황에서 막다른 화면 대신 일관된 안내와 복귀 경로를 제공.

## 2. 사용자 & 진입 경로 (상태별 트리거)
- **404 (NOT FOUND)**: 없는 URL, 삭제된 여행(`/trips/없는id`), `notFound()` 호출
- **에러 (500)**: 렌더링 중 예외, 서버 오류 → React error boundary 포착
- **403 (FORBIDDEN)**: 로그인했으나 멤버가 아닌 여행 접근(초대 필요)
- **점검 (MAINTENANCE)**: 배포/점검 모드 활성화 시
- **이탈**: 각 상태의 액션 버튼(아래) → 해당 목적지로 복귀

## 3. 구성 요소
> 레이아웃·색·간격은 시안 고정. 시안: `docs/design/prototype/시스템 페이지.dc.html`

**공통 레이아웃**
- 최소 상단 바(jero 로고만, 클릭 시 홈)
- 은은한 앰비언트 배경(상태 톤 글로우)
- **중앙 정렬 블록**: 일러스트(톤 원형 + 라인 아이콘 + 흰색 액센트 배지) → **코드 칩** → 제목 → 설명 → **액션 버튼** → (선택) 헬퍼 텍스트

**상태별 매핑**
| 상태 | 코드 칩 | 제목 | 액션 | 헬퍼 |
|---|---|---|---|---|
| **404** | `404 · NOT FOUND` | 길을 잃은 것 같아요 | 내 여행 목록으로(주) · 이전으로(보조) | — |
| **에러(500)** | `500 · ERROR` | 잠시 문제가 생겼어요 | 다시 시도(주) · 홈으로(보조) | 오류 코드(예: TRIP-5A21) |
| **403** | `403 · FORBIDDEN` | 접근 권한이 없어요 | 홈으로(주) · 초대 요청(보조) | — |
| **점검** | `점검 중 · MAINTENANCE` | 잠시 점검 중이에요 | 상태 페이지 보기(주) | 예상 완료 시각 |

## 4. 데이터
- **상태 설정(정적)**: 상태별 톤·아이콘·코드·제목·설명·액션 — 단일 출처 상수(`lib`)로 관리
- **동적 값**:
  - 에러: 오류 코드/추적 ID(로그 상관관계용 — 민감정보 비노출)
  - 점검: 예상 완료 시각(점검 설정에서)
  - 403: (선택) 어떤 여행인지 최소 정보 — 존재 노출 정책 §13
- 화면 내 사용자 입력 없음(읽기 + 액션만)

## 5. 상태
시안 상태 스위처(`404` / `error` / `403` / `maint`)에 대응 — 각 상태가 곧 화면.
- 상태 간 전환은 발생 원인(트리거)에 따라 결정되며, 한 화면에서 토글되는 값이 아님(시안의 스위처는 미리보기용).
- 공통 전환 애니메이션(fade) + 일러스트 float.

## 6. 인터랙션
1. **404** — "내 여행 목록으로" → `/trips` · "이전으로" → 브라우저 뒤로(history back)
2. **에러** — "다시 시도" → **error boundary reset**(재렌더 시도) · "홈으로" → `/trips`
3. **403** — "홈으로" → `/trips` · "초대 요청" → 소유자에게 접근 요청(메커니즘 §13)
4. **점검** — "상태 페이지 보기" → 상태/공지 페이지(목적지 §13)
5. 로고 클릭 → 홈

## 7. 권한 분기
- **403**은 인증된 사용자가 **멤버가 아닌 여행**에 접근했을 때. 비로그인 보호 라우트 접근은 401/로그인(01)으로 리다이렉트(시스템 페이지 아님).
- 권한 판정은 **서버에서**(§8.2). 클라 라우팅만으로 접근 제어하지 않는다.
- **보안 주의**: 403은 "여행이 존재함"을 드러낸다. 비공개성이 중요한 리소스는 404로 처리하는 정책과 충돌할 수 있음 → §13.

## 8. 라우팅 & 연결
- **구현 위치**(Next.js App Router):
  - 404 → `app/not-found.tsx` (+ `notFound()` 트리거)
  - 에러 → `app/error.tsx`(세그먼트 경계, `reset()` 제공) + `app/global-error.tsx`(루트)
  - 403 → 여행 라우트에서 비멤버 판정 시 표시(전용 컴포넌트 또는 `notFound()` 분기 — §13)
  - 점검 → 미들웨어/환경 플래그로 점검 페이지 표시
- **나가기**: `/trips`(홈) · 01(로그인, 401 경로) · 상태 페이지 · 브라우저 back
- 시안 내 네비게이션(참고): 로고/홈 액션 → 홈

## 9. 구현 메모
- **레이아웃 공유**: 4상태가 동일 레이아웃 컴포넌트(`SystemPage`)에 상태 설정만 주입. 중복 구현 금지 → `components`(또는 `features/system`).
- **Next.js 매핑**: `not-found.tsx`/`error.tsx`/`global-error.tsx` 특수 파일에서 `SystemPage`를 상태별로 렌더. `error.tsx`는 클라이언트 컴포넌트 + `reset` 사용. (Next 16 규약 — 불확실하면 `node_modules/next/dist/docs/` 확인, CLAUDE.md §2)
- **에러 처리**: error boundary가 예외 포착 → 사용자에겐 일반화 메시지 + 추적 ID, 상세는 서버 로깅(민감정보 비노출, §8.5).
- **점검 모드**: 환경 변수/플래그 기반. 미들웨어에서 점검 시 전 경로를 점검 페이지로.
- **디자인 토큰**: 08 에러 화면과 같은 중앙 일러스트 패턴. 기존 토큰 계승(배경/카드/프라이머리/Pretendard) — 하드코딩 금지(§7.1). 상태 톤(블루·앰버·퍼플·그린)도 토큰화.
- **접근성**: 적절한 상태 코드(404/500/403) 응답 + 스크린리더용 제목·랜드마크.

## 11. 수용 기준 (Acceptance Criteria)
> 이 화면이 "done"인 조건. GATE 1 승인·QA 테스트 시나리오의 근거.

- [ ] 없는 URL/삭제된 여행 접근 시 404 화면(코드 칩·제목·설명·"내 여행 목록으로"/"이전으로")이 시안대로 렌더된다.
- [ ] 렌더링 예외 발생 시 에러 화면이 표시되고, "다시 시도"가 error boundary를 reset해 재렌더를 시도한다.
- [ ] 로그인 사용자가 멤버가 아닌 여행에 접근하면 403 화면("초대 요청"/"홈으로")이 표시된다.
- [ ] 점검 모드 활성화 시 점검 화면(예상 완료 시각 포함)이 표시된다.
- [ ] 4개 상태가 동일 레이아웃에 톤·아이콘·문구·액션만 다르게 렌더된다.
- [ ] 각 액션이 올바른 목적지로 이동한다(목록/홈/뒤로/재시도/초대 요청/상태 페이지).
- [ ] 404/500/403에 대해 서버가 적절한 HTTP 상태 코드를 응답한다.
- [ ] 에러 화면에 민감 정보(스택·내부 메시지)가 노출되지 않고, 추적 ID만 표시된다(§8.5).
- [ ] 비로그인 보호 라우트 접근은 시스템 페이지가 아니라 로그인(01)으로 리다이렉트된다.

**비기능**: 데스크톱 1280px 기준 시안과 시각 일치 / 기존 토큰 계승 / 적절한 상태 코드·접근성(랜드마크·포커스) / 에러 화면도 앱 크래시 없이 안전하게 렌더.

## 12. 확정 사항 (MVP 범위 · 디자인·스펙 고정)
- **4개 상태**: 404 / 에러(500) / 403 / 점검.
- **공통 중앙 일러스트 레이아웃** + 상태별 톤·문구·액션 주입(단일 컴포넌트).
- **Next.js 특수 파일 매핑**: `not-found.tsx` / `error.tsx`(+`global-error.tsx`).
- **에러 일반화 + 추적 ID**(민감정보 비노출), 상세는 서버 로깅.
- 08 만료 링크 에러와 **동일 디자인 톤** 계승.

## 13. 열린 질문 (설계 단계에서 확정)
- **403 vs 404 정책**: 비멤버 여행 접근을 403(존재 노출)으로 보일지, 404(존재 은폐)로 처리할지 — 보안 vs UX 트레이드오프.
- **"초대 요청" 메커니즘**: 403에서 소유자에게 접근 요청을 보내는 흐름(MVP 포함 여부, 알림과 연계 — 알림 MVP 이슈와 연결).
- **점검 모드 트리거**: 환경 플래그 vs 원격 토글, 점검 페이지로의 라우팅 범위.
- **상태/공지 페이지**: "상태 페이지 보기"의 목적지(내부 페이지 vs 외부 status 서비스, MVP 포함 여부).
- **오류 추적 ID**: 생성·로깅 방식(에러 모니터링 도구 도입 여부).
- **점검 예상 시각**: 표시 데이터의 출처(점검 설정).

# 01. 로그인 / 랜딩

> 제품의 첫 진입 화면. 좌측은 제품 가치를 보여주는 랜딩, 우측은 로그인/회원가입.
> 비로그인 사용자는 공유 링크로 둘러보기까지 가능하다.

## 1. 개요
- **화면 ID**: `auth-landing`
- **라우트**: `/` (= `/login`. 로그인 상태면 `/trips`로 리다이렉트)
- **목적**: 신규/기존 사용자를 인증해 워크스페이스로 들여보내고, 비로그인자에게 제품 가치를 전달한다.

## 2. 사용자 & 진입 경로
- **사용자**: 비로그인 방문자(신규 가입 / 기존 로그인), 공유 링크 수신자
- **진입**: 도메인 직접 진입(`/`) · 보호 라우트 미인증 접근 시 리다이렉트 · 로그아웃 직후 · (이메일 인증/초대 링크)
- **이탈(성공 시)**: 로그인/가입 완료 → `/trips`(내 여행 목록)

## 3. 구성 요소
> 레이아웃·색·간격은 시안 고정. 여기서는 "어떤 요소가 있고 무엇을 하는가"만 정의. 시안: `docs/design/prototype/로그인 랜딩.dc.html`

**좌측 — 브랜드/가치 (랜딩)**
- 로고(핀 아이콘 + "jero"), 가치 배지("함께 만드는 여행")
- 헤드라인("친구들과 함께 짜는 여행 계획, 한 곳에서.") + 한 줄 설명
- 기능 하이라이트 3종: **실시간 협업 / 순서·동선 정리 / 예산·더치페이** (각 아이콘 + 제목 + 설명)

**우측 — 인증 패널**
- 모드별 타이틀/서브타이틀 (로그인: "다시 오신 걸 환영해요" ↔ 회원가입: "제이로 계정 만들기")
- **Google로 계속하기** (소셜 로그인 버튼)
- 구분선 "또는 이메일로"
- **이메일 폼**:
  - 회원가입 모드에서만 `이름` 필드 추가
  - `이메일`, `비밀번호`(가리기/보기 토글)
  - 로그인 모드에만 "비밀번호를 잊으셨나요?" 링크
  - 제출 버튼(로그인: "로그인" / 가입: "가입하고 시작하기")
- 로그인 ↔ 회원가입 **모드 토글**
- **공유 링크로 둘러보기** (비로그인 열람 진입점)
- 처리 중 **로딩 오버레이**(스피너 + 문구)

## 4. 데이터
- `User` (id, name, email, avatar) — 가입 시 생성, 로그인 시 세션 발급
- 입력 폼 값(클라 일시 상태): `{ name?, email, pw }` — 서버 전송 후 폐기, **저장/로깅 금지**(§8)
- 세션: 서버가 발급하는 인증 쿠키(HttpOnly) — 클라이언트는 토큰을 직접 보관하지 않음

## 5. 상태
시안의 상태 스위처(`idle` / `error` / `loading`)와 모드(`login` / `signup`)에 대응.
- **기본(idle)**: 빈 폼 입력 대기
- **검증 에러(error)**: 필드별 인라인 에러 메시지 + 에러 스타일
  - 이메일 형식 위반 → "올바른 이메일 주소를 입력해 주세요"
  - 비밀번호 6자 미만 → "비밀번호는 6자 이상이어야 해요"
- **처리 중(loading)**: 패널 위 오버레이(로그인: "로그인하는 중…" / 가입: "계정을 만드는 중…"), 중복 제출 차단
- **인증 실패(서버)**: 자격 불일치·중복 이메일 등 → 일반화된 에러 메시지(계정 존재 여부 노출 금지, §8.5)
- **이미 로그인됨**: `/`·`/login` 접근 시 `/trips`로 리다이렉트(화면 미표시)

## 6. 인터랙션
1. 모드 토글(로그인 ↔ 회원가입) → 필드/문구 전환, 에러 초기화
2. Google로 계속하기 → OAuth 플로우 → 성공 시 `/trips`
3. 이메일 폼 제출 → **클라 Zod 검증**(이메일 정규식, 비번 길이) → 통과 시 로딩 → 서버 인증 → 성공 시 `/trips`, 실패 시 에러 상태
4. 비밀번호 보기 토글 → 입력 마스킹 on/off
5. "비밀번호를 잊으셨나요?" → **비밀번호 재설정 플로우(MVP 포함)**: Supabase 재설정 이메일 발송 → 메일 링크 → 새 비밀번호 설정 페이지(신규 시안 없이 기존 인증 패널 컴포넌트·토큰 재사용)
6. 공유 링크로 둘러보기 → 읽기 전용 뷰어로 이동 (§8 참조)

> 클라 검증 규칙(시안 도출): 이메일 `/.+@.+\..+/`, 비밀번호 길이 ≥ 6. **클라 검증은 UX용이며, 신뢰 경계는 서버**(§8.3) — 서버가 재검증.

## 7. 권한 분기
- 이 화면은 **인증 이전**이라 owner/editor/viewer 권한 분기는 없다.
- 분기 축은 **인증 상태**: 비로그인(폼 노출) vs 로그인됨(리다이렉트).
- "둘러보기"로 들어온 비로그인 열람자는 **읽기 전용 뷰어 권한**으로만 동작(편집 UI 없음). 권한은 서버에서 강제(§8.2).

## 8. 라우팅 & 연결
- **들어오기**: `/`(직접) · 보호 라우트 미인증 리다이렉트 · 로그아웃 직후
- **나가기**:
  - 로그인/가입 성공 → `/trips`
  - 공유 링크로 둘러보기 → 뷰어 공유(읽기 전용). 단, 실제 뷰어 공유는 토큰 스코프(`/share/[token]`)가 필요 → 랜딩 버튼의 도착지는 §11에서 확정(데모 진입 vs 토큰 입력 안내)
- 시안 내 네비게이션(참고): Google/제출 → `내 여행 목록.dc.html`, 둘러보기 → `뷰어 공유.dc.html`

## 9. 구현 메모
- 폼: React Hook Form + Zod(`authSchema`: email·password, 가입 시 name). 시안 검증 규칙을 스키마에 반영.
- 라우팅: `app/(auth)/login/page.tsx` 또는 루트 `/` 처리. 로그인 상태 리다이렉트는 서버에서.
- 인증: Supabase Auth 후보(명세서 §8 미확정) — 소셜(Google) + 이메일. 세션은 HttpOnly·Secure·SameSite 쿠키(§8.4).
- 보안:
  - 보호 라우트는 서버에서 세션 검증(§8.2). 클라이언트 역할 신뢰 금지.
  - 인증 실패 메시지 일반화(계정 존재 여부 비노출, §8.5).
  - 가입·로그인 시도 rate limit 고려(§8.7).
  - 입력값 저장/로깅 금지, 비밀번호 평문 노출 금지(§8.5).
- 컴포넌트 매핑: `features/auth`(폼·소셜 버튼·모드 토글). 공통 입력/버튼은 `components/ui`(shadcn).
- 디자인 토큰: 시안의 색·radius·그림자는 `support.js`에서 추출해 `globals.css @theme`에 반영(하드코딩 금지, §7.1).

## 10. 디자인 시안 참조
- `docs/design/prototype/로그인 랜딩.dc.html` (모드: login/signup, 상태: idle/error/loading)

## 11. 수용 기준 (Acceptance Criteria)
> 이 화면이 "done"인 조건. GATE 1 승인·QA 테스트 시나리오의 근거. (체크리스트 = 검증 항목)

- [ ] 비로그인으로 `/` 진입 시 좌측 랜딩(브랜드/가치/기능 3종) + 우측 인증 패널이 시안대로 렌더된다.
- [ ] 로그인 상태로 `/`·`/login` 접근 시 폼을 보이지 않고 `/trips`로 리다이렉트된다.
- [ ] 로그인 ↔ 회원가입 토글 시 필드(가입 모드=이름 추가)와 문구가 전환되고, 기존 에러가 초기화된다.
- [ ] 이메일 형식 위반 / 비밀번호 6자 미만이면 필드별 인라인 에러가 표시되고 제출이 차단된다(클라 Zod).
- [ ] 유효 입력 제출 시 로딩 오버레이가 표시되고 중복 제출이 차단된다.
- [ ] 서버 인증 실패 시 계정 존재 여부를 드러내지 않는 일반화된 에러가 표시된다(§8.5).
- [ ] 로그인/가입 성공 시 `/trips`로 이동한다. Google 소셜 로그인도 성공 시 `/trips`로 귀결된다.
- [ ] 비밀번호 보기 토글이 입력 마스킹을 on/off 한다.
- [ ] 비밀번호 등 입력값이 클라이언트에 저장되거나 로깅되지 않는다(§8.5).

**비기능**: 데스크톱 1280px 기준 시안과 시각 일치 / 폼 전체를 키보드로 조작 가능(라벨·포커스 순서) / 인증 처리 중 사용자에게 진행 피드백 제공.

## 12. 확정 사항 (MVP 범위)
- **비밀번호 재설정**: ✅ MVP 포함. Supabase 재설정 이메일 → 새 비밀번호 설정 페이지. **신규 시안 없이** 기존 인증 패널 컴포넌트·토큰 재사용(디자인 고정 유지). 라우트는 `features/auth` 하위(예: `/reset-password`).
- **이메일 인증(가입 시 verification)**: ❌ MVP 제외.
- **"공유 링크로 둘러보기" 도착지**: ✅ **데모 여행**(읽기 전용 뷰어)로 진입. 토큰 입력 UI 없음.

## 13. 열린 질문 (설계 단계에서 확정)
- **인증 백엔드**: Supabase Auth 채택 여부(소셜·이메일·세션·RLS 일괄). 명세서 §8 미확정 — 설계의 `데이터모델_계약.md`에서 확정.
